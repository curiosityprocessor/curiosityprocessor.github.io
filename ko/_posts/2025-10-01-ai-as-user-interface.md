## 사용자 인터페이스의 변화

### CLI
사용자가 컴퓨터와 상호작용하기 위한 인터페이스 (UI)는 과거에는 CLI 형태였다.  
Command Line Interface (CLI)는 정확한 명령어와 옵션 값들을 모두 알고 있어야 했고,  
상호작용으로 반환되는 결과값도 줄글의 형태여서 피로감도 높았다.  
즉 일반적인 사용자가 접하기에는 진입 장벽이 높았고, 컴퓨터를 잘 아는 사람만 사용 가능했다.  

### GUI
그래픽 요소가 도입되면서 사람 친화적인 Graphical User Interface (GUI)가 등장했다.  
더는 정확한 명령어를 알 필요 없이, 눈으로 직접 실행하고 싶은 프로그램의 형태를 보고,  
클릭이나 엔터같은 단순한 액션으로 프로그램을 실행할 수 있었다.  
물론 초기에는 텍스트 입력, 날짜 입력 같은 여러 입력 컴포넌트와  
유효성 검증 실패 시 나타나는 에러문구나 팝업 등에 익숙해져야겠지만  
CLI에 비해 훨씬 진입장벽이 낮아져 일반적인 대중도 쉽게 배우고 사용할 수 있게 됐다.  

### AI의 등장
LLM을 기반으로 한 챗봇 형태의 프로그램이 대세가 되어가면서 채팅이 새로운 인터페이스가 되었다.  
특히 사용자의 의도를 파악해 작업을 자율적으로 수행하는 Agentic AI가 등장하면서  
사용자는 더 이상 텍스트, 날짜 같은 어떤 컴포넌트 값을 입력하는지 조차도 필요 없이  
텍스트 (또는 음성)으로 원하는 것을 자연어로 표현하면 그걸 알아서 해석해서 동작하는 형태가 가능해졌다.  
유저 인터페이스가 command line에서 graphic으로 넘어간 것처럼  
이제는 자연어, 더 정확히는 내포된 의도(intention)을 기반으로 한 인터페이스로 진화했다 볼 수 있다.  
ChatUI, NaturalUI, IntentionUI 등 아직 표현은 정돈되지 않은 것 같지만 차세대 인터페이스로 보는 관점이다.  

## 인터페이스 간 공존
GUI가 등장하고 폭발적인 인기를 얻었을 때에도, 모든 CLI 프로그램을 대체한 것은 아니었다.  
개발자 전용 프로그램이나 굳이 그래픽 요소가 불필요한 부분은 여전히 CLI로 왕성히 동작중이었고, 지금도 여전하다.  

AI도 마찬가지이다.  
"하나의 채팅으로 모든 걸 해결한다"는 생각은 이상적이고 기술적으로도, 서비스 관점에서도 부족한 점이 많다.  
LLM은 확률 기반 생성 메커니즘이기 때문에 아주 정확해야하는 값은 GUI로 입력 컴포넌트를 받는것이 정확하다.  
보안적인 측면이나 금융권, 의약 도메인에서 AI 활용이 제한되는 이유이다.  
일반적인 도메인에서도 입력값의 개수가 적거나, 입력 가능한 선택지가 적을 수록 GUI로 처리하는 것이 효율적일 수 있다.  
즉, AI UI는 GUI로 구현하기 불편했던 점, 어려웠던 점을 대신해야지 GUI가 이미 강점으로 잘하는 부분을 무차별하게 대체해서는 안된다.  

### AI UI로만 구현 시 문제점
예를 들어 아이디와 비밀번호를 받는 단순한 회원가입을 생각해보자.  
이걸 LLM 채팅으로 구현한다면, 사용자는 이런식으로 요청할 수 있다:  
"음.. ID는 abc 이걸로 하고 비밀번호는 def 아니다 def123 으로 해줘"  
그럼 LLM은 id: abc, pw: def123 같이 해석하여 처리하기를 기대할 것이다.  
그런데 적은 확률이라도 아이디를 'abc이걸'로 잡는다던가, 비밀번호를 'def'로 잡을 수도 있다.  
모델의 성능과 사용자 입력값에 따라 달라지겠지만, 요지는 확률적으로 발생할 수 있다는 점이다.  
이렇게 되면 LLM이 인식한 값을 다시 사용자에게 재차 확인시키는 프로세스를 추가해야 하고 오히려 구현 복잡도와 사용자 피로도를 높이게 된다.  
GUI가 잘하는 것, 즉 어떤 항목에 대한 입력값인지에 대한 인지와 정확한 값 전달은 GUI에게 맡기는 것이 낫다.  

또 다른 예로 설문조사를 들어보자.  
이 설문조사는 단순히 평가와 연령대만 받는다고 가정해보자.  
평가는 1~5점 사이를 선택해야하고, 필수값이다.  
연령대는 20대 부터 시작해서 10년 단위로 입력 받아야하고 최대값은 80+이고 선택 항목이다.  
이 설문조사 입력을 GUI가 아닌 AI UI로 구현한다고 생각해보자.  
사용자가 평가 항목에 1~5점이 아닌 "좋음", "나쁘지 않았어요", "별로였어요" 라고 대답한다면 이걸 어떻게 평가 점수화 할 것인가?  
연령대를 입력 불가한 10대 또는 정상적이지 않은 범위의 나이를 말한다면 어떻게 할 것인가?  
이 값은 유효하지 않은 값이라고, 유효한 값을 줄 때까지 반복적으로 주고받고를 해야하는가?  
비슷한 항목이 늘어난다면, 어떤 항목이 잘못 입력됐는지 AI가 항상, 정확하게 집어낼 수 있는가?  
위와 같이 고정된 값의 범위가 있거나, 유효성 검증 표현 또한 GUI에서 잘해내는 영역이다.  
이런 부분은 굳이 AI UI로 대체할 필요가 없다.  
기술의 발전은 과거 부족한, 불편한 점을 보완하는 것이지, 무리하게 모든 걸 대체하는 것이 아니다.  

### AI UI가 GUI보다 잘 할 수 있는 점

#### 입력단
AI가 기본적으로 잘하는 것은 패턴 인식이다. 다르게 표현하면 카테고리를 구분하는 것, 즉 라벨링이다.  
사용자의 입력값이 대략적인 의미만 맞으면 된다거나, 선택할 수 있는 항목이 매우 많아서 약간의 추상화하여 연결하는 용도로는 적합하다.  
예를 들어 회원가입 후 사용자의 취미를 받는 항목이 있다고 보자.  
GUI로, 특히 옵션으로 제공하게 된다면 선택할 수 있는 취미는 수 만가지가 될 것이다.  
그걸 사용자가 검색하여 선택하는 것도 피로도 높은 일이고, 원하는 값이 없을 수도 있다.   
그렇다고 임의의 텍스트 값을 허용하게 되면 추후에 그 데이터를 활용하는 서비스나 통계/분포 등에 활용하기 까다로워질 수 있다.  

AI는 이런 카테고리 분류 작업에 능숙하다.  
사용자는 자신이 원하는 값을 입력하고, LLM이 이걸 기존에 정의된 분류값과 가장 일치하는 항목과 연결하거나 역으로 선택 가능한 옵션을 제안할 수 있다.  

#### 출력단
AI가 잘하는 패턴 인식을 출력단에서 생각해보면, 어떤 값들을 어떻게 출력할지를 AI에게 위임할 수 있다.  
Agentic AI에서 특히 잘하는 분야인데, 목표를 설정해주면 그 목표를 어떻게 달성할지를 AI Agent가 자율적으로 수행하게 한다.  
보고서를 작성하는 서비스를 예로 들어보자.  
입력값은 이미 있다고 가정하고, 출력의 형태를 제어한다고 가정해보자.  
GUI로 구현한다면 데이터 범위를 어떻게 설정할지, 표를 그릴지 이미지를 추가할 지, 배치는 어떻게 할지 여러가지 옵션을 직접 설정해야 한다.  
커스터마이징 항목이 늘어날 수록 사용자가 조작해야 하는 값도 늘어나고 피로도도 증가하게 된다.  
나는 다음 분기 추이 보고서를 만들려고 들어왔는데, 세부 설정들을 조작하다보니 정작 내가 뭘 만들려고 하는지를 놓칠 수 있다.  

AI, 특히 Agentic AI는 이렇게 특정한 목표가 있고, 그걸 수행하기 위한 자잘한 설정들을 자동화하는데 활용할 수 있다.  
LLM을 기반으로한 생성형 AI의 최대 장점 중 하나는 학습한 수 만가지 데이터를 기반으로 어떤 형태의 글이든 작성 가능하다는 점이다.  
특히 이미지, 음성까지 입/출력 가능한 Multi-modal 모델의 경우 생성할 수 있는 범위가 훨씬 늘어난다.  
이런 장점들을 활용해 컨텐츠의 생산 측면에서 반복적이고 지루한 작업, 귀찮은 작업 등을 대체할 수 있다.  

물론 정밀한 제어가 필요할 때는 여전히 GUI는 유효하지만,  
일반적인 사용자가 대략적인 목표만 있고 구체적인 세부 설정까지는 모르거나 귀찮아하거나,  
예시나 가안을 생성하는 단계에서는 AI로 효율적으로 활용할 수 있다.  
이를 결합해 가안은 Agentic AI가, 이후 생성된 가안을 디테일하게 다듬는 건 GUI로 하는 하이브리드 UI를 제공할 수 있다.  


### 하이브리드 UI
AI와 GUI가 각자 잘하는 영역을 담당하는 하이브리드 UI가 점차 확대될 것이다.  
보고서 예시처럼 초안을 AI가, 디테일/검증은 GUI로 하는 방식으로 동일한 역할을 순차적으로 담당하는 방식도 있고,  
AI는 자연어로 입력받는 사용자의 의도를 분석해 사전에 정의된 여러가지 프로세스 중 어떤 것을 실행할지 결정하는 (컨트롤러/라우터) 역할을 하고,  
GUI는 구체적인 프로세스별로 정확한 값과 유효성 검증을 하는 역할 분담 방식이 될 수도 있다.  
결국 핵심은 어떤 것을 GUI가 잘하고, 어떤 것을 AI가 더 잘할 수 있는 지 분별하는 힘일 것이다.
